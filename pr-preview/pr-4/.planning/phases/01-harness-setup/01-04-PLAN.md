Body (30–80 lines) covers:

- **Status:** Skeleton. This skill will be filled in as Phase 2's installer code stabilizes. For now, it documents the intended shape.
- **Non-negotiable rules** (these are fixed — will not drift):
  - Every script starts with `#!/usr/bin/env bash` and `set -euo pipefail`.
  - Every script uses an `on_error` trap that logs the failing line and exits non-zero.
  - No `sudo npm install -g` anywhere. Always `sudo -u agent -H npm install -g <pkg>` via the `as_user` helper.
  - Idempotency is mandatory: `ensure_user`, `ensure_line_in_file`, `ensure_npm_prefix`, `ensure_symlink`. Never `useradd agent` without a pre-check.
  - Logging: `log_info`, `log_warn`, `log_error` (colored to stderr) from `plugin/lib/log.sh`.
- **Intended layout** (copy from HARNESS.md §1.1):
  - `plugin/bin/agentlinux-install` — entrypoint
  - `plugin/lib/{log.sh,idempotency.sh,as_user.sh,distro_detect.sh}`
  - `plugin/provisioner/{10-agent-user,30-nodejs,40-path-wiring,50-registry-cli}.sh` — ordered numbered scripts
- **Distro detection:** reads `/etc/os-release`; refuses to run on anything that is not `ubuntu` with `VERSION_ID` in `{22.04, 24.04}` (v0.3.0 scope).
- **PATH-wiring for all six invocation modes** (BHV-01..06): `/etc/profile.d/agentlinux.sh` for login shells; `~agent/.bashrc` for interactive; `~agent/.profile` for non-interactive SSH; `/etc/environment` augmentation for cron/systemd; explicit `Environment=PATH=...` in systemd unit templates; `/etc/sudoers.d/agentlinux` with `Defaults env_keep+=PATH` (mode 0440) for `sudo -u agent`.
- **Growth plan:** Phase 2 adds the real idempotency primitives; Phase 3 adds the Node.js installation pattern; this skill absorbs each as they land.

Body (30–80 lines) covers:

- **Status:** Skeleton. Grows with Phase 4. The stub `plugin/catalog/schema.json` was seeded in Phase 1; Phase 4 finalizes it.
- **Why this exists (CAT-03):** new agents must be addable by submitting *only* a JSON catalog entry and an install recipe — no CLI source changes. The schema is the contract that enforces that.
- **Current (Phase 1) schema shape** (mirror of `plugin/catalog/schema.json`):
  - Top-level: `{ version: string, agents: Agent[] }`.
  - `Agent`: `{ name: string (pattern ^[a-z][a-z0-9-]*$), description: string, install: string }`.

  - `install` becomes a reference to a filename under `plugin/catalog/agents/<name>/install.sh`.
  - Add an optional `remove` field referencing `plugin/catalog/agents/<name>/remove.sh` (symmetric uninstall — CLI-04).
  - Add a `defaults_to_installed: false` invariant-check so CAT-02 (no agents installed by default) cannot regress.
- **Install recipe contract** (what every `plugin/catalog/agents/<name>/install.sh` must do):
  - Be idempotent.
  - Source `plugin/lib/as_user.sh` and run npm / binary-download commands via `as_user <agent>`.
  - Never `sudo npm install -g`.
  - Emit clear log lines (`log_info`, `log_error`).
  - Return non-zero on any sub-step failure; do not swallow errors.
- **Validation:** pre-commit runs `plugin/cli/scripts/validate-catalog.mjs`. In Phase 1 this is a structural check; Phase 4 upgrades to ajv-based JSON Schema validation.
- **Growth plan:** Phase 4 finalizes the schema and adds the first three real catalog entries (claude-code, gsd, playwright). This skill absorbs the final shape at that point.

**B. `.claude/skills/qemu-harness/SKILL.md`** — frontmatter + placeholder body.

Frontmatter:
```yaml
---
name: qemu-harness
description: Use when running the QEMU-based behavior-test suite locally or debugging the nightly-qemu workflow. Documents the download + cache of Ubuntu cloud images, QEMU boot flow, SSH-into-guest pattern, bats-over-SSH execution, teardown, and how to add a new Ubuntu version. Grows with Phase 6's full harness implementation.
---
```

Body (30–80 lines) covers:

- **Status:** Skeleton. The real harness script lands in Phase 6; this skill documents the target shape so Phase 6 has a starting point and Phase 1's `nightly-qemu.yml` workflow knows what to invoke.
- **Why QEMU (not just Docker):** Docker false-positives (root-by-default, no systemd, minimal locale) mask real bugs. QEMU against a fresh Ubuntu cloud image catches what Docker misses. ADR-007 locks this decision.
- **Target flow** (`tests/qemu/boot.sh`):
  1. Download Ubuntu cloud image (`ubuntu-22.04-server-cloudimg-amd64.img` or `ubuntu-24.04-server-cloudimg-amd64.img`) into a local cache dir (default `~/.cache/agentlinux/qemu/`). Skip download if cached and SHA matches upstream.
  2. Generate cloud-init seed ISO (`tests/qemu/cloud-init/user-data` + `meta-data`) — creates a root SSH keypair and installs openssh-server.
  3. Boot QEMU (`qemu-system-x86_64` + `-cpu host` + `-m 2G` + `-smp 2` + cloud-image disk + seed ISO + port-forwarded SSH 2222).
  4. Wait for SSH (poll port 2222 until `nc -z localhost 2222`).
  5. scp the plugin tarball into the guest.
  6. Run installer over SSH.
  7. Run bats suite over SSH against the installed plugin.
  8. Shutdown the guest (`ssh root@localhost -p 2222 poweroff`) and `wait` on the QEMU process.
- **Adding a new Ubuntu version:** add a Dockerfile under `tests/docker/Dockerfile.ubuntu-<version>`, extend `boot.sh` with a new `--ubuntu <version>` branch, add a matrix entry in `.github/workflows/test.yml` and `nightly-qemu.yml`.
- **Local prerequisites:** `qemu-system-x86_64`, `cloud-image-utils` (`cloud-localds`), `openssh-client`, `curl`. Document `apt install` one-liner for Ubuntu hosts.
- **Growth plan:** Phase 6 writes the real `tests/qemu/boot.sh` and this skill absorbs the concrete flags and cloud-init template.
