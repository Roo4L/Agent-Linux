## Decisions Made

- **Literal-inlined 0440 in the install line (security-engineer preference).** The initial draft used `install -m "$SUDOERS_MODE"` with a `readonly SUDOERS_MODE="0440"` constant. After running the plan's acceptance grep `grep -qE 'install -m .0440.'` it failed (regex requires a char on each side of `0440`; bare space-0440-space had no char before `0`). Rather than relax the grep, I inlined the mode as a quoted literal `install -m "0440" -o root -g root ...` — both acceptance-grep-satisfying AND more auditable (a human scrolling the provisioner sees the mode directly on the install line, no indirection). Comment added noting the rationale.
- **`read -r -d ''` for content assembly (not printf-concat or ensure_marker_block).** The drop-in is a complete fresh file every install, not a block embedded in an existing file. A single-quoted heredoc captured into a variable via `read -r -d ''` gives me byte-stable content (no $-expansion) plus an easy printf write to the tmpfile. ensure_marker_block is the wrong primitive here — that's for partial edits to user-owned files.
- **`return 1` (not `exit 1`) in error paths.** This provisioner is SOURCED by plugin/bin/agentlinux-install's run_provisioners() loop. `return` hands control back to the caller so the entrypoint's ERR trap reports the correct src:line. `exit 1` would exit the parent process directly, losing the attribution. Matches the 30-nodejs.sh pattern (30-nodejs.sh line 70: "`return 1` — this provisioner is SOURCED, so `return` trips the entrypoint's strict-mode ERR trap with correct src:line attribution").
- **Bats uses `/opt/agentlinux-src/plugin/bin/agentlinux-install` (not `/agentlinux-src/...`).** The plan's inline bats sample used `/agentlinux-src/` but tests/docker/run.sh line 150 stages sources via `cp -R /workspace /opt/agentlinux-src`, matching tests/bats/10-installer.bats line 17 which uses `INSTALLER=/opt/agentlinux-src/plugin/bin/agentlinux-install`. Used the repo-canonical path.
