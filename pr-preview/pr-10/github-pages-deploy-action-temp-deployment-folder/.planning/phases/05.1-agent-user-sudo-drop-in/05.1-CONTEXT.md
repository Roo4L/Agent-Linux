<domain>
## Phase Boundary

Install `/etc/sudoers.d/agentlinux` granting the agent user passwordless sudo (scope: ALL commands per ADR-012). This unblocks Phase 5 recipes that need `apt install` / `systemctl restart` / etc (notably Playwright's `install-deps`) and aligns the project with the reality that real coding agents routinely need root.

Requirements in scope: INST-06 (agent has passwordless sudo), BHV-07 (drop-in integrity).

Out of scope: Phase 5 agent recipes (claude-code, gsd, playwright install bodies) — Phase 5.1 only provisions the sudo; Phase 5 uses it.

</domain>

<decisions>
## Implementation Decisions

### Sudoers Drop-In (per ADR-012)
- Path: `/etc/sudoers.d/agentlinux`
- Mode: `0440`
- Owner: `root:root`
- Contents: exactly one meaningful line (plus optional header comment) — `agent ALL=(ALL) NOPASSWD: ALL`
- Validation: `visudo -cf` must succeed before the file is installed. Fail-fast if invalid.
- Idempotency: re-runs produce byte-identical file (sha256 stable).

### Provisioner Shape
- New file: `plugin/provisioner/20-sudoers.sh`
- Runs AFTER `10-agent-user.sh` (agent user must exist — the sudoers line references `agent`)
- Runs BEFORE `30-nodejs.sh` and `40-path-wiring.sh` (order-independent in practice but numeric slot 20 signals "user-related setup, before Node")
- Uses lib primitives: `log_info`, `log_warn`, `log_error`, `ensure_dir /etc/sudoers.d 0755 root:root`, `visudo_validate` (from Phase 2 lib)
- Atomic write: write to tmpfile, `visudo -cf tmpfile`, `install -m 0440 -o root -g root tmpfile /etc/sudoers.d/agentlinux`, remove tmpfile
- Logs a one-line audit record: `log_info "Installed sudoers drop-in: agent user gets passwordless sudo (ADR-012)"`

### Test Coverage
- New bats file: `tests/bats/22-agent-sudo.bats` (slot 22 signals "agent-user-related, between BHV-01..06 in 20-agent-user.bats and RT-XX in 30-runtime.bats")
- `@test "INST-06: agent user has passwordless sudo"` — `run_sudo_u 'sudo -n true'` exit 0
- `@test "BHV-07: /etc/sudoers.d/agentlinux file integrity"` — mode 0440, owner root:root, passes `visudo -cf`, contains exact line
- `@test "BHV-07: sudoers drop-in is byte-stable on re-run"` — extends INST-02 sha256 set (or spot-check)
- `@test "INST-04: --purge removes sudoers drop-in"` — smoke that the file is gone after --purge (this is already implicitly covered by Phase 4 `--purge` bats but worth a dedicated assertion here)

### --purge Symmetry
- `plugin/bin/agentlinux-install --purge` already has a step that removes `/etc/sudoers.d/agentlinux` (was in Phase 4 Plan 04-06's teardown even though the file didn't exist yet — now it's load-bearing). No additional change needed.

<specifics>
## Specific Ideas

- **One provisioner, one bats file** — minimal scope. The plan is essentially "write 40 lines of bash + 20 lines of bats".
- **Security-engineer review matters here** — getting a sudoers drop-in wrong (wrong mode, not `visudo`-validated, syntax error) can break system sudo for everyone. The `visudo -cf tmpfile` gate before `install` is non-negotiable.
- **Log audit record is important** — users need to see in the install log that their agent now has sudo. One `log_info` line.
- **No surprise activation on re-run** — sha256 byte-stable. If ADR-012 changes scope later (e.g. to narrow allowlist), that's a separate ADR + revised drop-in.

</specifics>
