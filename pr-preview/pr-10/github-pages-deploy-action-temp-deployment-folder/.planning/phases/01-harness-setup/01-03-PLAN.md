### bash-engineer.md
- Rubric bullets: shellcheck compliance (run `shellcheck` via Bash if Bash is granted); idempotency primitives (`ensure_user`, `ensure_line_in_file`, `ensure_npm_prefix`) used instead of raw `useradd`/`echo >> file`/`export PATH=...`; POSIX-vs-bash correctness (we are bash, not POSIX — flagging `[[ ]]` as non-POSIX is a false positive); quoting ("$var" not $var around paths that may contain spaces); error propagation (`set -euo pipefail` at the top of every provisioner script; traps for cleanup); no `curl | sh` inside the installer without SHA verification.
- Gotchas: forgetting `set -o pipefail` (silent `|` failures); unquoted `$HOME` when running as a different user; using `eval` to expand PATH.
- Tools: `Read, Grep, Glob, Bash`.

### node-engineer.md
- Rubric bullets: Commander.js idiom compliance (subcommands via `.command()`, not manual `process.argv` parsing); TypeScript strict-mode compatibility (`strict: true` in tsconfig — no implicit `any`); error handling without swallowing (no bare `catch (e) {}`); `process.exit` used only at the top-level entry (never inside library functions); no `console.log` in library code (use a logger module); biome formatting.
- Gotchas: async functions that forget to `await`; `import` of `node:fs/promises` missing `await`; shelling out with `child_process.exec` instead of `execFile` (command-injection risk).
- Tools: `Read, Grep, Glob, Bash`.

### security-engineer.md
- Rubric bullets: `sudo` grants written to `/etc/sudoers.d/agentlinux` with mode `0440` (not 0644 — group-writable sudoers file is a privilege-escalation hole); NO `eval` in bash; NO `xargs -I {}` with untrusted input; word-splitting risks (variable expansion inside command substitution without quoting); curl-pipe-bash safety (release installer must SHA-verify the tarball before exec; `main(){...}; main` wrapping so a truncated download does not half-execute); input sanitization in catalog recipes (`name` field matches `^[a-z][a-z0-9-]*$`); secret leakage in logs (no echoing of environment variables or full command lines containing tokens).
- Gotchas: `chmod 755` on a sudoers drop-in silently breaks sudoers; `curl | bash` with no `set -e` in the installer body; `npm config set prefix` being run as root then expected to take effect for the agent user.
- Tools: `Read, Grep, Glob, Bash`.

### qa-engineer.md
- Rubric bullets: coverage of every BHV-01..06, RT-01..04, AGT-01..05, CLI-01..05, CAT-01..03, INST-01..05 (names may be in `.planning/REQUIREMENTS.md`); edge cases (pre-existing agent user; pre-existing Node.js from apt; partial prior install — installer interrupted halfway; dirty Docker image vs fresh Docker image); non-interactive shell paths covered by tests (cron, systemd `User=agent`, `sudo -u agent`, non-interactive SSH, interactive bash login — these are BHV-02..06); assertion quality (does the test assert something meaningful, or does it just `run` a command and discard output).
- Gotchas: a test that `runs` a command and checks exit code 0 without checking stdout (mutation-kill-weak); a test that only covers interactive shell and claims BHV coverage; a test skipped via bats `skip`.
- Tools: `Read, Grep, Glob, Bash`.

Body (50–120 lines):

## When to use this skill

Use it at the end of every non-trivial task where the agent modified code, tests, or reference docs. Skip it for: typo fixes, formatting-only changes, pure `.planning/` edits (GSD workflow state does not go through review).

## The loop

```
Main agent completes task
  │
  ├─ Look at what was produced (bash, TS, bats, docs, or mix)
  │
  ├─ Spawn appropriate review subagents in parallel:
  │   ├── Bash code changed → bash-engineer, security-engineer, qa-engineer
  │   ├── TS/JS code changed → node-engineer, security-engineer, qa-engineer
  │   ├── Bats tests changed → qa-engineer, behavior-coverage-auditor
  │   ├── Catalog recipes changed → catalog-auditor, security-engineer
  │   ├── Reference docs changed → (use global review-documentation)
  │   └── End of phase → behavior-coverage-auditor (TST-07 gate)
  │
  ├─ Collect free-form summaries from each reviewer
  │
  ├─ Triage: valid / irrelevant / contradictory / already addressed
  │
  ├─ Apply valid fixes
  │
  ├─ Re-spawn reviewers
  │
  └─ Stop when remaining comments are not actionable
```

## Dispatch rules

- Files matched by `^plugin/(bin|lib|provisioner)/.+\.sh$` → bash-engineer + security-engineer + qa-engineer.
- Files matched by `^plugin/cli/(src|test)/` → node-engineer + security-engineer + qa-engineer.
- Files matched by `^tests/bats/.+\.bats$` → qa-engineer + behavior-coverage-auditor.
- Files matched by `^plugin/catalog/(agents/.+/.+\.(sh|json)|catalog\.json|schema\.json)$` → catalog-auditor + security-engineer.
- Files matched by `^docs/.+\.md$` → global `review-documentation` (if available).
- End-of-phase: always spawn behavior-coverage-auditor to satisfy TST-07 regardless of what was changed.

## Triage rules (main agent keeps authority)

- **Fix:** a reviewer flags a concrete, verifiable bug (shellcheck warning, missing idempotency primitive, uncovered BHV-XX, sudoers mode 0644, etc).
- **Skip:** the reviewer asks for a stylistic change that contradicts another reviewer or is already addressed.
- **Stop:** remaining comments are stylistic preference, contradictory, or would require more work than the next phase's plan allocates.

The main agent owns the triage decision — reviewers are advisors, not gatekeepers.

## Trigger

Per CLAUDE.md and ADR-010: this skill is invoked by **CLAUDE.md instruction**, not a Stop hook. Stop hooks fire on every stop (user interrupts, context limits, errors) — not just task completion. A future lightweight Stop hook may run pre-commit + CLI unit tests for deterministic checks only (see HARNESS.md §4.4).

## Relation to TST-07

`behavior-coverage-auditor` is the acceptance gate for every phase from Phase 2 onward. TST-07 requires it to run at the end of every phase. This skill explicitly lists that requirement so the main agent does not forget to spawn it at phase-close.
