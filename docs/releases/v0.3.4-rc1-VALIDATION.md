# v0.3.4-rc1 — brownfield VM validation runbook

The maintainer acceptance gate for the v0.3.4 "Aware Installation Process"
release candidate. Run it on a **real brownfield VM** — an existing host with
prior agent tooling (an `agent` user, a system Node, a previously-installed
Claude Code) — not a throwaway container. The genuine acceptance: the aware
installer reuses or repairs what's already there without breaking it, and
`claude update` runs clean afterward.

Run everything as **root** (`sudo -i` if you are not already root) — the
installer refuses to run otherwise. Budget ~15 minutes; you can stop after any
step and roll back.

## 0. Snapshot first

Take a VM snapshot before touching anything — every later step is reversible by
restoring it.

## 1. Preview — what WOULD happen (zero mutation)

Fetch the installer and run it in `--dry-run`. It runs detection + decisions,
prints a pre-flight report, and exits 0 **without changing anything**:

```bash
export AGENTLINUX_VERSION=v0.3.4-rc1
curl -fsSL https://agentlinux.org/install.sh -o /tmp/al-install.sh
bash /tmp/al-install.sh --dry-run
```

The report has a `## DET-NN` section per component (install user, Node, npm
prefix, sudoers, catalog agents), each with a `✓/✗/•/—` status glyph and
`[DET-NN] key=value` detail lines. Read it against your host: a healthy `agent`
user and a NodeSource Node 22 read `✓` (they'll be reused); a Claude Code at a
non-canonical path or a wrong-owned npm prefix reads `✗`/`•` and prints a
`remediate: registered bail — component=… reason=…` warning (it'll need `--yes`
to repair). The structured decision line appears in step 2.

Then confirm nothing changed: `/etc/sudoers.d/agentlinux`, `/home/agent`, and
`/etc/passwd` are byte-identical to before the command.

## 2. Fail-safe — the installer won't mutate without consent

Run the real installer with **no** `--yes`. If any component needs a
state-overwriting repair, it bails **before mutating**, with a structured hint
and exit 65:

```bash
bash /tmp/al-install.sh ; echo "exit=$?"
```

Expect a line like
`[BAIL] component=npm-prefix reason=wrong-owner hint=run with --yes to chown or rebase`
and `exit=65`, and — again — nothing changed on disk. (If your host needs no
repairs it installs and exits 0; that's fine — skip to step 4.)

## 3. Real install — apply with consent

Re-run with `--yes` to authorize the repairs previewed in step 1:

```bash
bash /tmp/al-install.sh --yes ; echo "exit=$?"   # expect 0
```

Spot-check: `agentlinux list` shows the catalog; tools that were healthy were
reused, tools that were broken/mispathed were reinstalled at the canonical path,
and your `~/.claude/` data survived.

## 4. The canonical acceptance — `claude update` runs clean

If your host had no prior Claude Code (nothing for the installer to reuse or
repair), install it first: `agentlinux install claude-code`. Then run the
self-update — the canonical bug AgentLinux exists to eliminate:

```bash
sudo -u agent -H bash --login -c 'claude --version && claude update && claude --version'
```

Confirm: exit 0, the version does not go backwards (post ≥ pre), and the output
has **zero** `EACCES` / `permission denied` lines.

## 5. Rollback (if anything looks wrong)

Restore the step-0 snapshot. Without a snapshot: `agentlinux-install --purge`
removes the agent user, `/home/agent`, and the PATH artefacts (it keeps Node);
a pre-existing system npm prefix outside the home is left intact.
