# Pitfalls Research — v0.5.0 Agenda Redefinition

**Domain:** OSS product-strategy authoring + landing-page refresh after a positioning shift
**Researched:** 2026-05-09
**Confidence:** HIGH (strategy-doc + JTBD + landing-page categories cross-checked against multiple sources; AgentLinux-specific application is grounded in PROJECT.md + index.html + STABILITY-MODEL.md)

## Scope

v0.5.0 broadens AgentLinux from one pillar (separated agent environment, shipped) to three pillars (separated env + stability/benchmarks + security hardening), where pillars 2 and 3 are FORWARD-LOOKING and land in v0.6+. Three deliverables:

1. Canonical product-strategy doc (likely `docs/strategy/`)
2. ADR-015 capturing the framing decision
3. Website refresh at agentlinux.org

Most failure modes for this milestone are **rhetorical** (what the doc claims vs. what is delivered), **structural** (doc layout, downstream-surface drift), or **propagation** (site copy says one thing, doc says another). This file catalogs them with prevention rules concrete enough to become phase-plan acceptance criteria.

**Phrasing-rule glossary used throughout** (cite verbatim from downstream phases):

- **Delivered-fact voice** — present-tense indicative, names a shipped behaviour, links to a `@test`, an ADR, a CI gate, or a release artefact. Examples: "AgentLinux installs an agent user with a per-user npm prefix (AGT-02, Ubuntu 22.04 + 24.04 + 26.04)." / "The release pipeline blocks on TST-08."
- **Forward-looking voice** — first-person plural commitment with an explicit horizon and no present-tense product subject. Examples: "We are committing to publish a benchmark harness in v0.6+." / "Our roadmap commits to a supply-chain threat model before any pillar-3 mitigation lands." NEVER write "AgentLinux benchmarks…" or "AgentLinux defends against…" for unshipped behaviour.
- **Aspirational drift** — any sentence that uses delivered-fact voice for an unshipped behaviour. The single most dangerous pattern in this milestone.

## Strategy-Document Pitfalls

| # | Pitfall | What it looks like | Why it happens | Warning sign | Concrete prevention |
|---|---------|--------------------|----------------|--------------|---------------------|
| 1 | **Vague vision statement** | "Make Linux great for agents." / "Be the best agent environment." Could describe any project; no falsifiable claim; no decision-influencing content. | Author writes the vision last, after the energy is spent on pillars; defaults to a slogan. | Vision sentence still applies after you swap "AgentLinux" for "Devbox" / "Nix" / "mise". | Acceptance check: vision sentence MUST contain (a) a named user category, (b) the mechanism (separated environment + curated combos + security posture), (c) a specific outcome (e.g., "self-update without `sudo`/`EACCES`"). Reviewer rubric question: "If a competitor copy-pasted this vision into their README, would it still describe them?" If yes → reject. |
| 2 | **JTBD that's actually a feature list** | Bullets like "user can install agents", "user can pin versions", "user can run `agentlinux upgrade`". Each maps 1:1 to an existing CLI verb. | Working backward from existing features to retrofit jobs. The Christensen Institute calls this the most dangerous JTBD pitfall ([source](https://www.christenseninstitute.org/theory/jobs-to-be-done/)). | Every "job" begins with "user can …" or aligns 1:1 with a `agentlinux <verb>` subcommand. | Acceptance check: each job MUST be solution-agnostic (no AgentLinux-specific noun), name the user's situation/struggle, and survive the substitution test ("could a non-software solution do this job?"). Format rule: "When `<situation>`, I want to `<motivation>`, so I can `<outcome>`." Phase reviewer must reject any job whose `<motivation>` clause names a CLI verb or AgentLinux feature. Reference exemplar: re-phrase pillar-1 from "user can install Claude Code" to "When I provision a fresh Linux host for an agent, I want the agent toolchain installed under an identity that can self-update, so I never have to debug `EACCES` mid-session." |
| 3 | **"Everyone" target users** | "AgentLinux is for developers who want to run AI agents." Persona section enumerates 5+ archetypes with no priority. | Fear of excluding any potential user. Treats positioning as marketing rather than constraint-setting. | Persona section has no rejected/de-prioritized personas; word "everyone", "anyone", "developers" appears unqualified. | Acceptance check: persona section MUST name (a) one **primary** persona with a specific role + employer-type + present-tense problem, (b) one **secondary** persona, (c) one **explicitly de-prioritized** persona with the reason. AgentLinux candidate primaries (research-grounded): (i) infra/platform engineers at companies operating coding-agent fleets (Cloudlinux-shaped), (ii) AI-tool builders shipping agent-based products who need a reproducible target environment, (iii) coding-agent-curious developers running Claude Code / Cursor on their own boxes, (iv) OSS maintainers contributing to AgentLinux itself. Pick one as primary; the others are secondary or non-targets. Reviewer rubric: "Could the strategy doc reject a feature request because it serves the de-prioritized persona?" If no → personas are too broad. |
| 4 | **Listing pillars without prioritization** | Three pillars presented as a numbered list with equal-length paragraphs and identical voice. No statement of which earns the next milestone, which is opportunistic, which is foundational. | Author optimizes for narrative symmetry over decision utility. Strategy literature consistently flags this: "good strategic goals are specific enough to exclude things" ([Wanger](https://jennywanger.com/articles/stop-scoring-everything-how-to-prioritize-with-strategy/)); equal-weighted pillars provide no prioritization signal. | Three pillars look interchangeable; reviewer cannot answer "which pillar do we sacrifice first under resource pressure?" from the doc alone. | Acceptance check: pillars section MUST include a "**Pillar priority**" subsection that explicitly tags each pillar with one of: `foundational` (don't regress), `next-milestone` (we're committing to ship in v0.6+), `opportunistic` (we'll pick this up if external contribution or time allows). For AgentLinux this lands as: pillar 1 = `foundational`, pillar 2 OR pillar 3 = `next-milestone`, the other = `opportunistic`. The strategy doc owner must commit to which is which; deferring the choice fails the gate. |
| 5 | **Three-pillar trap (atrophy)** | One pillar (almost always pillar 1, since it's shipped + tested) dominates over time; the other two pillars get one paragraph each in v0.5.0 and never get updated. By v0.7 the doc reads as a single-pillar story with two vestigial appendices. | Shipped surface generates evidence (tests, releases, ADRs) that demands documentation; unshipped pillars don't. The asymmetry compounds. | At any milestone close, pillar 2 + pillar 3 sections have not been edited since v0.5.0; pillar 1 section has grown materially. | Acceptance check (process): the strategy doc has a per-pillar "**Last reviewed**: vX.Y.Z" header; `/gsd-complete-milestone` checklist requires re-reading every pillar section and updating the date or noting "no change required, evidence: …". Doc owner role: assigned per pillar (not per doc) so each has a named accountable contributor. Quarterly review item even when no milestone closes. |
| 6 | **Overpromising forward-looking work (aspirational drift)** | "AgentLinux benchmarks token consumption against vanilla setups." / "AgentLinux defends against prompt-injection attacks." Both sentences use delivered-fact voice for unshipped behaviour. The single most dangerous v0.5.0 failure mode — directly produces vaporware perception ([source](https://en.wikipedia.org/wiki/Vaporware)). | Author wants the strategy doc to feel ambitious and confident; present-tense indicative reads stronger than future-tense; the asymmetry between shipped pillar 1 and unshipped pillars 2/3 collapses unintentionally. | Any sentence in pillar 2 or pillar 3 sections has subject = "AgentLinux" + verb in simple present, OR uses "provides" / "offers" / "ensures" / "protects" / "defends" without a link to a `@test` / ADR / CI gate / shipped artefact. | Acceptance check (automated grep): in the strategy doc, for any line under a pillar-2 or pillar-3 heading, `grep -nE '^[^a-z]*AgentLinux (provides|offers|ensures|protects|defends|benchmarks|measures|hardens|isolates|detects|prevents)\b'` MUST return zero matches. Replacement phrasing rule: "We are committing to `<behaviour>` in `<milestone>`." / "Our roadmap commits to `<behaviour>` before `<milestone>` ships." / "AgentLinux today does not yet `<behaviour>`; v0.6+ adds `<behaviour>`." Reviewer rubric: every claim about an unshipped behaviour MUST appear in a sentence whose grammatical subject is "we", "our roadmap", or an explicit milestone identifier — never "AgentLinux". |
| 7 | **Mixing direction with progress** | A pillar section contains a paragraph of "we will" mixed mid-flow with a paragraph of "we have"; reader cannot tell which sentences describe shipped code and which describe direction. | Author writes the pillar as a single narrative arc instead of separating voices. | Pillar section paragraphs alternate freely between past/present-perfect (shipped) and future ("we will"). | Acceptance check (structural): every pillar section MUST be split into exactly two named subsections — `### Today (delivered, vX.Y.Z)` and `### Direction (forward-looking)` — with a horizontal rule between them. The `Today` block uses delivered-fact voice and links to evidence; the `Direction` block uses forward-looking voice and links to roadmap themes / tracking issues / ADRs-to-be-written. Pillar 1's `Direction` block is allowed to be "no further direction; this pillar is foundational and tracked via the bats suite" — but the subsection MUST exist for symmetry. Comparable OSS exemplars worth grepping for tone: rg's NOTES + DESIGN docs (clear separation); mise's "Coming Soon" / "Status" tags ([source](https://github.com/jdx/mise)). |
| 8 | **Conflating product strategy with technical roadmap** | Strategy doc reads as a list of features with timelines: "Phase 1: agent user. Phase 2: catalog. Phase 3: benchmarks." Indistinguishable from `.planning/ROADMAP.md`. | Author conflates "what we'll build" (roadmap) with "why we exist + who we serve + what we won't do" (strategy). Common when strategy doc is treated as a marketing artefact rather than an internal decision-making artefact. | Strategy doc has phase numbers, dates, or task lists; could substitute for ROADMAP.md without loss. | Acceptance check: strategy doc MUST contain (a) **Vision**, (b) **Target users / personas**, (c) **JTBD**, (d) **Three pillars (with Today/Direction split per pitfall #7)**, (e) **Positioning vs alternatives**, (f) **Non-goals (per pitfall #9)**, (g) **Roadmap themes appendix** (themes only, no phase numbers, no dates). Forbidden in body: phase numbers, sprint identifiers, story points, dates more specific than `vX.Y+`. The roadmap-themes appendix is the only place forward-looking work is named, and it links *out* to ROADMAP.md / Jira rather than restating it. |
| 9 | **No "what we will NOT do" section** | Doc has only positive claims; no explicit non-goals. Over time, every feature request looks plausibly in-scope; doc loses prioritization power. ProductTeacher: "without non-goals, everyone walks away with a different opinion of what the strategy is" ([source](https://www.productteacher.com/articles/implementing-non-goals)). | Author finds it culturally awkward to say no in writing; non-goals feel negative. | "Non-goals" / "What we won't do" / "Out of scope" heading absent from doc; only PROJECT.md has them. | Acceptance check: strategy doc MUST contain a top-level **Non-goals** section with ≥5 specific entries, each tagged with the pillar or general scope it bounds. Each entry has a one-sentence rationale. Format: "We will not `<thing>` because `<reason>`." Required AgentLinux candidates to consider (pick the live ones): "We will not become a distro again", "We will not ship sandboxing inside the agent environment in v0.5/v0.6", "We will not optimize pillar 2 benchmarks for non-coding agents in v0.6", "We will not bundle a default agent (no opt-out installs)", "We will not support Windows or macOS as targets", "Pillar 3 will not attempt to defend against malicious agents the user has explicitly authorized." Reviewer rubric: "Has the team rejected at least one feature request in the last quarter that this section would justify rejecting?" If no → non-goals are too abstract. |
| 10 | **Authorial drift (voice inconsistency)** | Doc opens with confident "we" voice ("We believe agents need their own user"); later sections drift into passive ("It is recommended that…") or third-person ("AgentLinux's approach is…"). | Multiple authors over time, or one author copying paragraphs from ADRs/READMEs that use different voice. | Single section contains both "we" and "the project" / passive voice; word "we" usage frequency drops sharply mid-doc. | Acceptance check: pre-merge `grep -cE '\bwe\b|\bour\b' docs/strategy/*.md` ≥ 1 per 200 words across the entire doc; reviewer reads aloud check: "If I read any randomly-selected paragraph, can I tell who is speaking?" Style guide line: "First-person plural ('we', 'our') throughout. Passive voice forbidden in vision, JTBD, pillars, non-goals; permitted only in the positioning section when comparing alternatives neutrally." |
| 11 | **Picking a framework and then half-applying it** | Lean Canvas with three boxes filled and six empty. Vision Board with no Target Group. Press-Release-First template with a press release that's actually a feature changelog. | Author commits to a framework for legitimacy but the framework's harder boxes (Unfair Advantage, Channels, Cost Structure) require research the milestone didn't fund. | Doc has visible framework scaffolding (named sections matching a known framework) with empty / TODO / one-line entries. | Acceptance check: every section heading from the chosen framework MUST be present AND have ≥3 sentences of substantive content (not "TBD", not "see Phase X", not a single noun). Phase plan must explicitly enumerate the chosen framework's required sections at planning time; phase-close gate fails any section under 3 sentences. Reviewer rubric: "Could a reader unfamiliar with the framework infer the section's purpose from its content alone?" |
| 12 | **The strategy doc never updates again** | Doc lands in v0.5.0 phase A, gets celebrated in the release notes, and is never edited. By v0.7 it describes pillars 2/3 as "forward-looking" even though v0.6 shipped them. The doc becomes a museum piece. | OSS projects culturally treat docs/decisions/ as immutable; strategy doc inherits that mental model when it shouldn't. | Strategy doc's `git log` shows zero commits between milestone closes; "Today" section per pillar still references v0.5.0 after v0.6 ships. | Acceptance check (process binding): the `/gsd-complete-milestone` template gains a mandatory step "Strategy doc reviewed; pillar `Today` sections updated for any newly-shipped behaviour; pillar `Direction` sections updated to remove now-shipped commitments." Strategy doc gains a top-of-file "**Last reviewed:** `<date>` at `<milestone close>`" header that the milestone-close gate enforces. Optionally: a CI lint that fails if the header date is older than the most recent release tag by >90 days. |
| 13 | **Confusing strategy with marketing copy** | Doc reads like landing-page hero copy: superlatives, no rejected alternatives, no concrete trade-offs, no admission of weakness. Useless internally because it doesn't help the team make decisions. | Author writes the doc with the website refresh in mind and conflates the audiences. Strategy doc's audience is the team + serious contributors, not casual visitors. | Doc contains marketing verbs ("revolutionize", "empower", "unlock"); no section named "Trade-offs", "Rejected alternatives", or "Known weaknesses"; reads identically whether read by a maintainer or a first-time visitor. | Acceptance check: strategy doc MUST contain a **Trade-offs / rejected alternatives** subsection within each pillar (or a dedicated section), naming ≥1 alternative seriously considered and ≥1 concrete trade-off accepted. AgentLinux candidate trade-offs to surface: pillar-2 — benchmark fidelity vs. CI cost (we're committing to coarser cheaper benchmarks first); pillar-3 — comprehensive sandboxing vs. agent productivity (we accept that giving the agent NOPASSWD sudo per ADR-012 widens attack surface in exchange for the workflow); pillar-1 — Ubuntu-only vs. broader distro support (deferred). Reviewer rubric: "Does this doc help the team say no to something? Or does it only help us say yes?" |

## Website-Refresh Pitfalls (post-positioning-shift)

The current `index.html` (v0.1.0, 1,045 LOC, single-file static, terminal aesthetic) currently markets AgentLinux as **"a purpose-built Linux distribution that runs on a dedicated machine"** with hero CTA "Join the waitlist". This framing is **already two pivots stale** (v0.2.0 distro retired 2026-04-18; v0.3.0 ships an installable plugin; v0.4.0 went OSS; v0.5.0 broadens to three pillars). Every pitfall below cites a concrete location in the current page that needs to change.

| # | Pitfall | What it looks like | Why it happens | Warning sign | Concrete prevention |
|---|---------|--------------------|----------------|--------------|---------------------|
| 14 | **False-advertising the broadening (the central risk)** | Landing page lists pillar 2 (benchmarks) and pillar 3 (security) as features alongside pillar 1, with verb forms that imply they exist today. Visitor installs AgentLinux, finds no benchmark harness, no security mitigations; trust collapses. | Marketing instinct to make all three pillars look equally substantive in the visual hierarchy. | Any pillar-2 or pillar-3 feature card uses present-tense delivered-fact voice ("Measures token consumption", "Defends against prompt injection") instead of forward-looking voice. | Acceptance check (visual + textual): every pillar-2 / pillar-3 section on the landing page MUST carry a visible status badge (e.g., `[v0.6+ ROADMAP]` / `[COMING SOON — v0.6+]`) at parity with how pillar 1 is left unbadged or carries `[SHIPPED v0.3.0]`. Textual check identical to strategy-doc pitfall #6: `grep -nE 'AgentLinux (benchmarks|measures|defends|protects|prevents|hardens)\b'` on the rendered HTML must return zero matches; allowed phrasing is "Coming in v0.6+:" / "Our roadmap:" / "We're committing to:". ## Top 5 Most Likely to Bite Us Specifically

Tuned to AgentLinux's situation — small team, two of three pillars are ambition, public repo since v0.4.0, existing landing page predates two pivots, strong harness culture but no prior strategy-doc culture.

1. **Pitfall #6 — Aspirational drift (overpromising forward-looking work).** *Highest risk.* The whole milestone exists because we want pillars 2 and 3 to be part of our story before they're shipped. The temptation to use delivered-fact voice is structural, not accidental. Plus we've gone OSS in v0.4.0 — mis-claiming a benchmark or security capability now reaches the public, not just internal stakeholders. Mitigation MUST be the automated grep + the subject-noun rule, not just reviewer vibes. This is the one that, if we get wrong, lands AgentLinux in a HN comment thread about vaporware.

2. **Pitfall #14 — False-advertising the broadening on the website.** *Same root cause as #6, much higher visibility surface.* The current landing page is already over-promising (it sells a distro that doesn't exist). The site-refresh phase has historically been done lightly (v0.1.0 was 2 days; subsequent pivots barely touched the site). Without an explicit acceptance criterion forcing roadmap badges and forward-looking-voice on every pillar-2/pillar-3 surface, we will ship a beautiful page that misleads visitors into expecting capabilities we don't have.

3. **Pitfall #22 — Strategy doc lands without updating README + CONTRIBUTING + ROADMAP.** *Highest probability.* Our small team has a track record of landing docs in `docs/` and forgetting README.md (the README still names v0.3.0 as the current state at line 1; PROJECT.md dates document drift between phases). The strategy doc at `docs/strategy/` is the *most likely* doc to suffer this fate because it deliberately doesn't replace the README — it complements it — so the README rewrite feels optional. It isn't.

4. **Pitfall #4 — Listing pillars without prioritization.** *Most likely structural failure.* Three pillars + nothing-yet-shipped on two of them + small team that doesn't want to "pre-decide" the v0.6 milestone topic = strong incentive to write all three pillars in equal voice with equal length. We will need to commit to one of pillar 2 / pillar 3 as `next-milestone` and the other as `opportunistic` BEFORE the doc lands, even though that decision feels premature. The acceptance check (must explicitly tag each pillar) forces the call.

5. **Pitfall #12 — Strategy doc never updates again.** *Most likely long-term failure.* AgentLinux has strong harness culture (bats suite, ADR convention, phase-close gates) but no living-doc culture. ADRs are immutable by convention; PROJECT.md drifts between updates (last-updated dates lag). Without a binding entry in `/gsd-complete-milestone`, the strategy doc will calcify and we'll be debating a v0.7 doc shaped by v0.5 reality. This is also the pitfall that makes pitfalls #6 and #14 worse over time: aspirational-voice claims that were honest in v0.5 become misleading once pillars start shipping but the doc doesn't move.

The first four are mitigated by acceptance criteria the v0.5.0 phase plans can directly cite from this file. The fifth is mitigated only by amending the milestone-close convention itself, and so requires a deliberate process change (not just an in-milestone check) — flag it for the v0.5.0 retrospective and the `/gsd-new-milestone` template update that follows.
