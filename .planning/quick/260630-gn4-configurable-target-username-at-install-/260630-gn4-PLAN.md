<objective>
Make the AgentLinux installer provision under an operator-chosen target username
instead of always provisioning the hardcoded `agent` user, while keeping `agent`
the default for greenfield installs.

Purpose: brownfield maintainers already have a working login (e.g. `nikita`,
`claude`); forcing a fresh `agent` user + manual file migration blocks adoption
on existing VMs. This implements Jira AL-50 and closes the AL-59 "alt-user
hollow-install" bug (an accepted alternate name today only partially provisions
— the npm prefix, PATH wiring, sudoers, and catalog CLI still land on `agent`).

Output: a fully threaded resolved-user variable across the entrypoint, every
provisioner, the sudoers drop-in, and the catalog CLI; a hardened username
validator (rejects root/reserved/system accounts); an interactive prompt; and a
behavior-test suite covering all five acceptance criteria.

Scope guards (do NOT implement — from the AL-50 "Out of scope" list):
- In-place rename of an already-installed AgentLinux user.
- Multi-user installs (more than one agent user on the same host).
</objective>
