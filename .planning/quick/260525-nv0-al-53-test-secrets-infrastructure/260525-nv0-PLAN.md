      ```bash
      # tests/bats/helpers/secrets.bash — AL-53 test-secrets contract.
      #
      # Design invariants (mirror helpers/assertions.bash):
      #   - No `set -euo pipefail` at top: sourced by bats via load.
      #   - require_secret references the named var via bash indirect expansion
      #     `${!var_name-}` (NOT eval, NOT `${!var_name}` without default —
      #     the dash form returns empty under `set -u` if the var is unset).
      #
      # Usage in a @test:
      #   require_secret ANTHROPIC_API_KEY
      #   # ... test body uses $ANTHROPIC_API_KEY ...

      require_secret() {
        local var_name=$1
        if [[ -z ${!var_name-} ]]; then
          skip "${var_name} not provisioned; see docs/internals/test-secrets.md"
        fi
        return 0
      }
      ```
    Hard constraints (security-engineer + bash-engineer will check):
      - Variable name comes from `$1`, NOT from `$@` or any concatenation.
      - Use `${!var_name-}` (dash-default form) so the helper works under
        `set -u`. The plain `${!var_name}` form errors out under nounset.
      - NO command substitution on the var name (`eval`, `printenv "$var_name"`
        via `$()`). Indirect expansion `${!var_name-}` is the safe path.
      - `skip` is a bats builtin — calling it from a `load`ed helper inside
        a `@test` body is supported and produces a yellow TAP line.

      ```
      <Two short paragraphs:
       1. Naive approach: drop the key into a fixture file or pass it as

          and both invisible to reviewers six months later.
       2. The class of tests that need secrets is small but growing.

          them will get the leak path wrong.>

      <Describe the four-layer pipeline in prose:
        1. `.env.local` at repo root (developer's private values).
        2. `.env.local.example` (committed template; lists supported vars).
        3. `tests/docker/run.sh` sources `.env.local` and forwards only

           sandbox key — green stays green.
        2. **Explicit allowlist beats env-file.** A two-row bash array in

           four lockstep edits, documented in the worked example below.
        4. **Sandbox keys are rotation-friendly.** A live sandbox key with
           90-day rotation cost is cheap; documenting the rotation
           procedure means future-you doesn't relearn it.
      ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer host shell -> Docker container | `.env.local` values forwarded via `-e VAR` cross from a long-lived developer env into a short-lived test container. |
| GitHub Actions repo secrets -> workflow env | `secrets.ANTHROPIC_API_KEY` exposed to a specific job step's process tree via the workflow runner. |
| Bats test process -> $output / logs | Once forwarded, the secret is in the test process env; any `printf '%s' "$ANTHROPIC_API_KEY"` or accidental log line could leak it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-AL53-01 | Information disclosure | `.env.local` committed to git | mitigate | `.env.*` already in `.gitignore` (line 37) AND SEC-05 gitleaks pre-commit gate AND full-history gitleaks job in `test.yml`. `.env.local.example` contains only commented placeholders (no `KEY=value` uncommented pairs — verified by automated grep in Task 1). |
| T-AL53-02 | Information disclosure | `docker run -e "VAR=$VAR"` interpolation | mitigate | Use the `-e VAR` form (no `=value`), Docker reads the value from the calling shell's env via the daemon. Avoids the secret appearing in any other process's `ps` output. Reviewed in Task 2 hard constraints. |
| T-AL53-03 | Information disclosure | `--env-file` blanket forward | mitigate | Explicit `SECRET_ALLOWLIST` bash array in `tests/docker/run.sh`; `--env-file` rejected by name in Task 2 hard constraints AND the automated `! grep -q '\-\-env-file'` check in verify. |
| T-AL53-04 | Information disclosure | Per-PR CI receives release-gate secret | mitigate | `test.yml` is explicitly NOT edited. Only `nightly-qemu.yml`'s boot step gets the env. Verified by `! grep -q 'ANTHROPIC_API_KEY' .github/workflows/test.yml` in Task 2 verify. PR authors cannot exfiltrate the secret via a malicious test, because per-PR jobs never see it. |
| T-AL53-05 | Information disclosure | Job-level `env:` over-broadcasts | mitigate | Step-level `env:` block on the `QEMU boot + installer + bats` step only — not job-level. Narrower blast radius. Documented in Task 2 hard constraints. |
| T-AL53-06 | Tampering | Variable-name injection in `require_secret` | mitigate | `${!var_name-}` indirect expansion; no `eval`, no command substitution on the var name. A malicious var name like `$(rm -rf /)` would be a literal lookup of a parameter with that name, returning empty — no shell interpretation. |
| T-AL53-07 | Information disclosure | Bats `$output` capture leaks secret on assertion failure | accept | Bats's diagnostic block prints `$output` on failure. If a test calls `run printf '%s' "$ANTHROPIC_API_KEY"`, the secret would land in `$output`. Mitigation is the test-author contract (documented in test-secrets.md "What AgentLinux does": never echo the secret) — not enforceable at the helper level. Accept for AL-53; AL-54 will exercise the live key and revisit if a real leak path appears. |
| T-AL53-08 | Repudiation | Sandbox-key rotation cadence missed | accept | The 90-day rotation procedure is documented in `test-secrets.md`. There is no automated calendar reminder under this ticket (avoid-ceremony rule — single developer, single key today). Future ticket can add a reminder if the secret count grows. |
| T-AL53-09 | Elevation of privilege | Sourced `.env.local` runs arbitrary bash | accept | `.env.local` is by definition the developer's own file in their own checkout. A compromised developer machine already has the keys to the kingdom; sourcing a file they wrote does not raise the threat ceiling. `set -a` / `set +a` bracketing limits the side effects to env exports. |
</threat_model>
