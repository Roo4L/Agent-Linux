<objective>
Land the bulk of Phase 15: the `--dry-run` flag on BOTH the bash entrypoint and the CLI install subcommand, the TTY per-action prompt loop that hooks into the Phase 14 DECIDE-THEN-ACT pipeline, and the Sentinel widening that captures user declines. After this plan, every state-overwriting Remediate (REMEDIATE-01 chown, REMEDIATE-03 sudoers-drift, REMEDIATE-04 reinstall-broken) is gated by a TTY-only `Proceed? [Y/n]` prompt; declining one prompt skips THAT remediation, marks the component `reused-with-warning` in the sentinel, and continues the install. `--dry-run` runs the full Phase 12-14 detect → decide pipeline and exits 0 without mutating any state.

**The architectural insertion point this plan establishes (extends Phase 14 DECIDE-THEN-ACT):**

```
parse_args                                # --dry-run / --yes / --no-yes; EX_USAGE on contradictions (D-15-04)
detect::run_once                          # read-only (Phase 12 — unchanged)
remediate::collect_all_decisions          # populate RESOLUTIONS + BAILED_COMPONENTS; ZERO mutation (Phase 14 — unchanged)
                                          # NEW: also populate ACTION_MAP[<component>]=<action-token> so prompt loop knows which action token to render

if DRY_RUN_REQUESTED:                     # NEW (D-15-01)
  detect::emit_report "$REPORT_FORMAT"    # bails surface IN report
  exit 0                                  # always 0 — preview semantic

remediate::flush_bails_or_continue        # Phase 14 — unchanged; exits 65 on bail in non-TTY/no-yes path

if [[ -t 0 ]] && [[ "$YES_FLAG" != true ]]: # NEW (D-15-05, D-15-10)
  prompt::run_all                         # NEW (D-15-06): asks per-action prompt per state-overwriting component;
                                          # declines → RESOLUTIONS[<component>]=reuse-with-warning + DECLINED_COMPONENTS[<c>]=<reason>;
                                          # additive actions never prompt (D-15-09)

run_provisioners                          # Phase 14 — provisioners now ALSO honor reuse-with-warning (skip mutation; emit warning)
```

**Why the prompt loop runs AFTER flush_bails_or_continue:** the Phase 14 bail-gate is for the NON-TTY path. In TTY mode, BAILED_COMPONENTS is empty (Phase 14's gate_or_bail only registers a bail when YES_FLAG=false AND state-overwriting AND non-TTY) — but Plan 15-01 must ALSO make Phase 14's gate consult TTY-presence. The implementation: extend `remediate::gate_or_bail` to a new branch — when state-overwriting AND no --yes AND TTY: do NOT register bail; instead record into ACTION_MAP for prompt.sh to consult. flush_bails_or_continue then no-ops in TTY mode, and prompt::run_all takes over.

**Sentinel widening (D-15-02):** `reused-with-warning` is a new terminal state for a component that was a Remediate candidate but the operator declined the fix. The component is left as-is (no mutation), the sentinel records the decline reason, and `agentlinux list` surfaces this prominently so the operator does not forget there is a known-defective component AgentLinux is now adopting.

**CLI `--dry-run` (TS half):** The CLI subcommand `agentlinux install <name> --dry-run` runs loadCatalog + tryReuse + tryRemediate decision + dryRun report + exit 0. Parallels the bash entrypoint's behavior for the per-agent install path (which is a separate operator invocation per Plan 14-03's T-14-12 design).

This plan owns UX-01 (--dry-run on both halves) + UX-02 (TTY per-action prompts with decline-and-continue + sentinel widening). Plan 15-02 owns UX-04 (alt-user flow), the phase-close AUDIT, and the REQUIREMENTS.md flip.

Purpose: Phase 14 landed the mutation contract (non-TTY bail-or-yes); Phase 15-01 lands the operator-facing TTY UX on top of it. The prompt loop is the most-visible v0.3.4 feature — when a maintainer-VM operator runs `agentlinux install` interactively, they get an itemized "should I fix this?" per-action confirmation, NOT a wall of automated mutations. The dry-run preview gives the same operator a non-mutating preview before they commit.

Output: A working `--dry-run` on both bash + CLI halves, a working TTY per-action prompt loop with decline-and-continue, sentinel widening with decline_reason, list-suffix rendering, and a `tests/bats/15-preflight-ux.bats` covering the foundation contract end-to-end (including byte-equal snapshot tests proving non-mutation on dry-run, expect/script-driven TTY simulation tests proving the prompt loop). Plan 15-02 layers UX-04 alt-user on top and closes the phase.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| argv (--dry-run / --yes combos) → parse_args | Operator argv can be contradictory; we MUST reject not silently pick a winner |
| TTY detection ([[ -t 0 ]]) → prompt-loop entry | A script could redirect stdin from /dev/tty making us think interactive — by design, behavior is "use TTY path" (D-15-05 contract) |
| Prompt response (read -r) → decline → sentinel | A user typing extra characters could attempt to embed shell control sequences; we mitigate via read -r -n 1 (no escapes parsed) |
| decline_reason token → sentinel JSON | The token comes from a fixed enum (chown-declined, sudoers-drift-declined, reinstall-broken-declined); never user-supplied |
| Component name → sentinel JSON via jq --arg | Component names come from RESOLUTIONS keys (hardcoded literals: user, npm-prefix, sudoers, agents.<catalog-id>); catalog ids are validated by Phase 4 regex |
| Provisioner reuse-with-warning case-branch → skip mutation | A bug in the case-branch could fall through to mutation despite the decline — covered by no-mutation snapshot tests |
