1. Extend `detect::user_probe()`: AFTER the existing `home_writable` assignment block (after `fi` on line ~51), add a new `can_sudo_apt` probe. When `present=true`, compute via:
   ```bash
   if sudo -u "$user" -n /usr/bin/apt-get --help >/dev/null 2>&1; then

     can_sudo_apt=false
   fi
   ```
   When `present=false`, set `can_sudo_apt=false` in the else-branch.

   CRITICAL — T-13-02 mitigation: use the absolute path `/usr/bin/apt-get` (NOT bare `apt-get`). A user with NOPASSWD for `/usr/bin/apt-get` who has a malicious shim earlier on PATH could pass a bare-`apt-get` probe without actually having the sudo grant the probe claims to verify. Do NOT use `as_user` here — `as_user` uses `sudo -u ... -H -E --` which adds `-H` and `-E` flags that change sudo's behavior; the probe needs the raw `sudo -u <user> -n` shape so a passwordless-sudo-failure surfaces as exit 1 rather than a password prompt.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Detection cache → REUSE decision | `/run/agentlinux-detect.json` is writable by root during install. REUSE consumes the cache via the `detect::*` reader functions (which read DETECT_* env exports, NOT the file directly). The cache exists only for cross-process re-runs of `--report-only`, not for in-process REUSE decisions; in-process state lives in exported bash variables. |
| User → REUSE-01 sudo bar | The `detect::user_can_sudo_apt` probe spawns `sudo -u <user> -n /usr/bin/apt-get --help`. The user could mount a `/usr/bin/apt-get` shim if they have write access to that directory (which would require root, so this is a defense-in-depth concern, not a privilege escalation). |
| Existing user state → AgentLinux ownership | A REUSE-1 decision adopts an existing system user as the "agent" user. Phase 14 will then potentially install the sudoers drop-in for them. The REUSE-01 sudo-bar requirement (NOPASSWD-for-apt) is the gating contract. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering | `/run/agentlinux-detect.json` cache | mitigate | The entrypoint calls `detect::run_once "$INSTALL_USER"` UNCONDITIONALLY before any `reuse::*` call (verified by reading `plugin/bin/agentlinux-install` line ~445). `detect::run_once` always overwrites the cache file. REUSE decisions consume the in-process DETECT_* exports, NOT a re-read of the cache file. Treating the cache as a transient memoization (not a trust boundary) means a tampered cache file present before installer invocation is overwritten before any REUSE call could see it. Verify via `awk` ordering check in Task 2's automated verify: detect.sh source line precedes reuse.sh source line; `detect::run_once` call precedes any provisioner that calls `reuse::`. |
| T-13-02 | Elevation of Privilege | `detect::user_can_sudo_apt` probe | mitigate | The probe uses the absolute path `/usr/bin/apt-get` (NOT bare `apt-get`). A user with NOPASSWD for `/usr/bin/apt-get` but a malicious shim earlier on PATH cannot trick the probe — `sudo` resolves the argument literally when it's an absolute path. Verified via `grep -F '/usr/bin/apt-get' plugin/lib/detect/user.sh` in Task 1's automated verify. Additionally, the probe uses `sudo -u <user> -n` (NOT `as_user` which adds `-H -E` that could carry caller env), so the probe's environment is exactly what sudo provides (subject to sudoers `secure_path`). |
| T-13-03 | Tampering | Reuse-on-incompatible-user (predicate bypass) | mitigate | The five REUSE-01 predicates (CONTEXT.md Area 1 Q1) all hold for the REUSE branch; any single predicate failing falls through to `bail` (irreconcilable: shell, home_writable, name-mismatch) or `remediate` (fixable: sudo_apt — Phase 14 picks up). The decision function `reuse::user_decision` returns the token via stdout; the case-branch in 10-agent-user.sh is the dispatcher. This is the install transcript on a host the operator is provisioning; if the operator can read /var/log/agentlinux-install.log they already have the host's /etc/passwd.
