         detected_source: "pre-existing",  // future Phase 14 may classify finer-grained

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `/run/agentlinux-detect.json` cache → CLI REUSE decision | The CLI install.ts reads this file (root-writable) and uses its contents to decide whether to skip dispatchRecipe. A tampered cache could trick install.ts into adopting a malicious binary as the catalog agent. |
| Sentinel JSON shape → CLI list/upgrade/remove | Sentinel files at `/opt/agentlinux/state/installed.d/<id>.json` are root-written by the CLI but agent-readable. A confused script or malicious agent (which doesn't have write to /opt/agentlinux/state by default but could escalate via REMEDIATE-04 reinstall pathway in Phase 14) could potentially tamper. |
| `agentlinux remove` aggressive ownership → user's manually-installed binaries | After REUSE adoption, `agentlinux remove <agent>` will DELETE the user's manually-installed binary. The user may not have realized adoption happened. |

## STRIDE Threat Register

Note: Plan 13-02 uses threat IDs T-13-05..T-13-08 to avoid collision with Plan 13-01's T-13-01..T-13-04 (which cover bash-side cache + sudo-apt-probe + predicate-bypass + marker-disclosure risks). Cross-plan threat IDs are globally unique within Phase 13.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-05 | Tampering | `/run/agentlinux-detect.json` cache (consumed by install.ts) | mitigate | The CLI install.ts is invoked as the `agent` user (via the entrypoint's `as_user agent /opt/agentlinux/cli/.../dist/index.js install <name>` path). The detect cache is root-written and on tmpfs; agent CAN read but cannot write. The cache is overwritten on every `agentlinux-install` run (Plan 13-01 / Task 2 — entrypoint calls detect::run_once unconditionally; see Plan 13-01 T-13-01 for the bash-side ordering proof). The CLI install.ts does NOT re-run detection itself; it consumes the cache produced by the most-recent installer invocation. Acceptable trust model: cache is as trustworthy as the root-owned installer that wrote it. Verified via grep in Task 2 verify chain: hardcoded `/run/agentlinux-detect.json` path + env override `AGENTLINUX_DETECT_CACHE` only used by tests. |
| T-13-06 | Tampering | Inherited `detect::user_can_sudo_apt` PATH-shim mitigation | mitigate (transitive) | Plan 13-01 T-13-02 establishes the bash-side mitigation: the user-can-sudo-apt probe uses absolute path `/usr/bin/apt-get` (NOT bare `apt-get`). | T-13-07 | Tampering | Sentinel reader trusts JSON shape blindly | mitigate | `agentlinux list/upgrade/remove` re-validate the binary's existence + ownership BEFORE acting on a `reused` sentinel. Specifically: `removeCmd` uses `existsSync(sentinel.binary_path)`; if false, skips dispatchRecipe and only deletes the sentinel. `upgradeCmd` runs a `validateReusedBinary` check; on missing binary, classifies as drift and forces reinstall (treating it as an un-installed entry). A malicious agent (or confused script) that writes a `status: "reused"` sentinel for a non-existent binary cannot trick the CLI into executing arbitrary uninstall.sh paths against missing files. A user who runs `agentlinux remove claude-code` expecting it to be a no-op on a reused install GETS what `list` told them would happen. Verify via Task 3 list-suffix @test. |
