# Phase 22 Plan 01 — QEMU EL9 release-gate harness + pipeline wiring

**Goal:** prove the EL9 port on a real AlmaLinux 9 QEMU guest and gate the v0.3.5
tag on EL9 Docker + QEMU green. QEMU is unavailable locally → author here,
validate in CI; flip gates to HARD only after CI proves the EL9 QEMU row green.

## Tasks

1. **boot.sh family dispatch** (`tests/qemu/boot.sh`)
   Generalize Ubuntu-only → `{22.04,24.04,26.04,almalinux-9}`: TARGET/FAMILY/RELEASE
   dispatch; IMG_NAME from the URL basename; family-selected cloud-init seed.
   SSH stays `root@` both families (AlmaLinux GenericCloud ships `PermitRootLogin
   yes`; only the root password is locked), so the installer + bats run as root —
   byte-equivalent to the Ubuntu arm.

2. **Checksum guard — HARN-02** (`tests/qemu/boot.sh`)
   `verify_one_checksum` requires the pinned image's line to exist and verifies
   only it (`sha256sum --check --strict`), closing the `--ignore-missing`
   zero-match false-pass (≥1 row matched). `selftest_checksum_guard` flips a byte
   in a synthetic file and asserts the verify FAILS, on every run. **Host-tested:
   5/5 (intact OK, corruption rejected, missing-line rejected, ext4/xfs anchor).**

3. **cloud-images.txt** — pinned dated `AlmaLinux-9-GenericCloud-9.8-20260526.x86_64.qcow2`
   + its `CHECKSUM` (GNU format — same `sha256sum` path as Ubuntu).

4. **EL9 cloud-init seed** (`cloud-init/user-data.almalinux9`) — root pubkey;
   `runcmd: dnf install -y epel-release; dnf install -y bats; systemctl enable
   --now sshd` (bats is EPEL-only and can't co-install with epel-release in one
   `packages:` transaction; `cloud-init status --wait` blocks until runcmd done).

5. **CI wiring** (`nightly-qemu.yml`, `release.yml`)
   nightly-qemu: matrix `ubuntu`→`target` incl `almalinux-9` (full arm — the green
   signal). release.yml gate-3-qemu: add `almalinux-9` as EXPERIMENTAL
   (continue-on-error) so a red EL9 leg doesn't skip gate-4 / red releases.
   gate-2-docker almalinux-9 stays experimental for now.
