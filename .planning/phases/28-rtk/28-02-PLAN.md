<objective>
Create the reusable, sourced shell helper `plugin/catalog/lib/prebuilt-binary.sh` that encapsulates the prebuilt-binary install machinery: architecture detection, download of the release asset + `checksums.txt`, a gzip magic-byte precheck, `sha256sum -c` verification BEFORE extraction, extraction of a named binary, install into `~/.local/bin`, and a version-lock assertion. This is the ENABLE-01 keystone — written once here so Phase 29 (gh) and onward are a catalog-entry + thin-recipe change with zero further CLI source edits (CAT-03 spirit).

Purpose: Centralizing verify-before-extract (the security keystone, RESEARCH.md Pattern 1 + §Common Pitfalls 1-2-3) prevents each future recipe from re-implementing — and getting wrong — the curl→verify→extract dance. The provisioner stages this file automatically: `50-registry-cli.sh` already copies the whole catalog tree with `cp -R "$CATALOG_SRC"/.`, so the new `lib/` subdir rides to `/opt/agentlinux/catalog/<ver>/lib/` with NO provisioner edit.
Output: a sourceable (non-executable) helper exposing well-named functions that rtk's install.sh (Plan 03) and future binary recipes call.
</objective>

<!-- The repo's existing verify-before-extract discipline to mirror. -->
packaging/curl-installer/install.sh uses: `curl -fsSL`, a gzip magic-byte check
(`head -c 2 … | od -An -tx1` → expect `1f8b`, because `file(1)` is not on minimal
images), then `sha256sum -c`. Mirror that discipline exactly.

<!-- Verified asset facts (RESEARCH.md §Code Examples, live 2026-06-30). -->
checksums.txt format is the standard two-space form: `<sha256>␣␣<asset-name>`.
sha256sum -c resolves the filename column relative to CWD → download under the
exact upstream asset name and run the check from inside the tmpdir.

    2. `al_pb_detect_asset(<tool>)` — echo the per-arch asset name via
       `case "$(uname -m)"`: `x86_64) echo "<tool>-x86_64-unknown-linux-musl.tar.gz"`;
       `aarch64|arm64) echo "<tool>-aarch64-unknown-linux-gnu.tar.gz"`; `*) al_pb_die

       Pitfall 3.)
    3. `al_pb_fetch_and_verify(<base_url> <asset> <tmpdir>)` — `curl -fsSL
       "${base}/${asset}" -o "${tmp}/${asset}"` and `curl -fsSL
       "${base}/checksums.txt" -o "${tmp}/checksums.txt"`, each `|| al_pb_die`.
       Then gzip magic guard: `magic=$(head -c 2 "${tmp}/${asset}" | od -An -tx1 |
       tr -d ' \n'); [[ "$magic" == "1f8b" ]] || al_pb_die "...not gzip
       (magic=${magic:-empty})"`. Then select ONLY the asset's line and verify:
       `grep -E "  ${asset}\$" "${tmp}/checksums.txt" > "${tmp}/${asset}.sha256"
       || al_pb_die "no checksum line"`; `( cd "$tmp" && sha256sum -c
       "${asset}.sha256" ) >/dev/null 2>&1 || al_pb_die "SHA256 verification
       failed for ${asset} — aborting BEFORE extract"`. The function must return
       non-zero (so the recipe aborts) on ANY failure, and must NOT extract.

    Quote every variable. Use the exact two-space `grep -E "  ${asset}\$"` form
    (checksums.txt is the standard two-space format — RESEARCH Pitfall 2).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub release CDN → local filesystem | Untrusted bytes (asset + checksums.txt) are downloaded over the network before any extraction or install |
| extracted tarball → ~/.local/bin | Archive contents (permissions, paths) cross into an on-PATH agent-owned dir |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-03 | Tampering | downloaded asset | mitigate | `sha256sum -c` against the release `checksums.txt` BEFORE extract; helper returns non-zero (recipe aborts) on mismatch — verify-before-extract enforced and grep-checked in acceptance criteria |
| T-28-04 | Tampering/DoS | 404/redirect HTML written to the tarball path | mitigate | `curl -fsSL` (mandatory `-f`) + gzip magic-byte (`1f8b`) precheck before sha256 (V5) |
| T-28-05 | Spoofing | wrong-arch / wrong-tool binary | mitigate | Fixed per-arch asset enum (`case "$(uname -m)"`, die on unknown) + version-lock assert (`--version` contains the pin) |
| T-28-06 | Elevation | forged owner/permissions inside the tarball | mitigate | `tar --no-same-owner` + `install -m 0755` sets explicit mode; install target is agent-owned `~/.local/bin` (no root, no /usr/local shim, V4) |
| T-28-07 | Tampering | checksums.txt itself substituted | accept | Both asset and checksum come from the same GitHub release over TLS; pin + TLS are the trust anchors. Pinning the checksums.txt digest is out of ENABLE-01 scope (RESEARCH §Security) |
</threat_model>
