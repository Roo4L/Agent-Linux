# Phase 6 Plan 06-03 / Phase 22: cloud image manifest for tests/qemu/boot.sh.
# Format: <target-key> <image-url> <checksums-url>
#
# boot.sh reads this file at startup and looks up the row for the requested
# TARGET key (Ubuntu rows key on the bare version `22.04`; the AlmaLinux row
# keys on `almalinux-9`). It downloads <image-url> into $AGENTLINUX_QEMU_CACHE,
# then always re-downloads <checksums-url> and runs `sha256sum --check` on the
# pinned image's line (Pitfall 10 — verify on EVERY cache hit, not just on
# first fetch). Both manifests are GNU coreutils format, but Ubuntu's
# SHA256SUMS emits BINARY-mode lines (`<hash> *name`) while AlmaLinux's CHECKSUM
# emits TEXT-mode lines (`<hash>  name`); boot.sh's verifier matches the
# filename field exactly (via awk, not a fixed-separator regex), so one
# verification path handles both.
#
# The .github/workflows/nightly-qemu.yml `actions/cache` step keys on
# hashFiles('tests/qemu/cloud-images.txt'). Rotating any URL here therefore
# invalidates the CI cache automatically — no manual key bump needed.
#
# One row per supported target. Add a row (and a matrix entry in the workflow +
# a Docker mirror) when a new distro becomes a support target.
#
# ALMALINUX PIN CAVEAT: AlmaLinux removes superseded dated builds from the
# images dir (keeps only `-latest` + the newest dated build), so a pinned older
# URL eventually 404s. Treat a download 404 on the almalinux-9 row as the bump
# signal — refresh to the current `AlmaLinux-9-GenericCloud-<minor>-<YYYYMMDD>`
# build (and its CHECKSUM is the same URL, regenerated in place).
